Question:
- If you create a constructor method to build a DVD by title then create a constructor method to build
a DVD by category. Does JAVA allow you to do this?

Answer: It’s not allowed in JAVA. Title and Category have the same data type (String), and Java differentiates constructors based on the number, type, and order of parameters—not by the parameter names. Because both constructors would accept exactly one String argument, they would share the exact same method signature. The Java compiler cannot determine which constructor to execute when you pass a string value.

Question 1: After the call of swap(jungleDVD, cinderellaDVD) why does the title of these two
objects still remain?
Answer: Because object references are passed by value. The swap method only swaps local copies of the references, leaving the original objects and their titles unchanged.

Question 2: After the call of changeTitle(jungleDVD, cinderellaDVD.getTitle()) why is
the title of the JungleDVD changed?
Answer: Because while the method receives a copy of the reference, that copied reference still points to the exact same object in memory as the original jungleDVD variable. When the changeTitle method uses that reference to modify the object's internal data (like calling a setTitle() method), it is mutating the actual object itself. Since the original jungleDVD variable points to that same object, it reflects the updated title.

Question 3: Is JAVA a Pass by Value or a Pass by Reference programming language?
Answer: JAVA is a pass by value programming language.

Question: Which classes are aggregates of other classes? Checking all constructors of whole classes if they initialize for their parts?
Answer: Based on the document, the Cart class is an aggregate of Media objects. The Store class is also an aggregate of Media objects. The CompactDisc class is an aggregate of Track objects. Additionally, the Book class aggregates a list of authors (String). The constructors for these "whole" classes (Cart, Store, CompactDisc, and Book) must initialize their respective collection fields (e.g., instantiating a new ArrayList) so that parts can be added to them without triggering a null pointer exception.
Question: If the passing object is not an instance of Media, what happens? (In reference to overriding the equals(Object o) method)
Answer: If the passed Object obj  is not an instance of Media, directly casting it to a Media object will throw a ClassCastException at runtime, causing the program to crash. To prevent this, you should verify that the object is of the correct type using the instanceof keyword before performing the cast.
Question: Iterate through the list and print out the information of the media by using toString() method. Observe what happens and explain in detail.
Answer: When looping through the ArrayList<Media> and executing m.toString() on each element , the program demonstrates polymorphism at the behavior level. Even though the reference variable is of the abstract type Media , Java dynamically determines the actual object type at runtime (e.g., Book, CompactDisc, or DigitalVideoDisc). It then invokes the specific overridden toString() method belonging to that exact subclass, printing the uniquely formatted information for that specific media type.
Question: Suppose we are taking this Comparable interface approach. What class should implement the Comparable interface?
Answer: The abstract Media class should implement the Comparable<Media> interface. Because it is the common superclass for Book, CompactDisc, and DigitalVideoDisc, implementing it here allows any two media items in the cart to be compared with one another.
Question: In those classes, how should you implement the compareTo() method be to reflect the ordering that we want?
Answer: You would override the compareTo(Media other) method to define a single comparison logic. For instance, to sort by title then cost, the method would first compare this.getTitle() with other.getTitle(). If the titles evaluate as identical, the method would then compare this.getCost() with other.getCost() to break the tie.
Question: Can we have two ordering rules of the item (by title then cost and by cost then title) if we use this Comparable interface approach?
Answer: No. The Comparable interface only allows a class to have a single "natural" ordering because the compareTo() method can only be overridden once. If an application requires multiple, distinct sorting sequences, you must utilize the Comparator interface instead.
Question: Suppose the DVDs has a different ordering rule from the other media types, that is by title, then decreasing length, then cost. How would you modify your code to allow this?
Answer: You would modify the sorting logic to check the runtime type of the objects being compared. Using the instanceof operator, you check if both objects are DigitalVideoDisc. If they are, you cast them and apply the DVD-specific sorting rules (compare by title, then by length in descending order, then by cost). If the objects are not both DVDs, the code would fall back to the standard sorting logic defined for general Media objects.
